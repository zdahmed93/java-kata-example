Critical Weaknesses

- Unused import fr.carrefour.kata.model.PlanType in both CreateSubscriptionRequest and ChangeSubscriptionRequest.
- The proration formula is mathematically questionable: it subtracts the unused value of the old plan from the new plan price, which can produce a negative nextCycleWithProrated. No guard against this.

### 2. Spring Architecture & Design Patterns — 3.5/10

- DTOs exist for requests (Records).

Critical Weaknesses

- Packaging is by technical layer (controller, service, dao, model, dto, event) rather than by domain/feature. Acceptable for a small kata but worth noting.
- ISubscriptionService uses the I prefix convention (Hungarian notation), which is not idiomatic Java. The standard is SubscriptionService (interface) and SubscriptionServiceImpl or a more descriptive name.
- The SubscriptionEventPublisher component is commented out (//@Component) and its injection in the service is also commented out. Kafka — a mandatory requirement — is effectively not integrated.

Critical Weaknesses

### 4. Persistence & Performance — 3/10

- init.sql provides a proper schema with foreign key constraints and seed data.
- MySQL is used via Docker, not H2.

Critical Weaknesses

- No pagination (Pageable) — no list endpoint exists at all.
- SubscriptionDao has no custom queries. A findByCustomerIdAndStatus or similar would be expected for real business logic.
- The init.sql schema is used alongside ddl-auto=update, meaning the schema is managed in two places simultaneously.

## Top 3 Prioritized Refactorings

1. Fix the broken build and wire Kafka (Blocker)
The test dependencies (spring-boot-starter-data-jpa-test, spring-boot-starter-webmvc-test) don't exist — the project likely doesn't compile cleanly. Simultaneously, Kafka — a mandatory requirement — is entirely commented out. Both the SubscriptionEventPublisher component and its injection must be uncommented and made functional. This is the most critical gap between the kata requirements and the submission.

2. Stop leaking JPA entities through the API layer
The controller returns raw Subscription entities. The SubscriptionResponse DTO exists but is never used, and it itself embeds Customer and SubscriptionPlan entities. A mapper (MapStruct is already declared) must be created to project the entity into a clean DTO. This also eliminates the LazyInitializationException risk during JSON serialization of lazy associations.

3. Add a global exception handler and return correct HTTP status codes
There is no @RestControllerAdvice. IllegalArgumentException thrown for "not found" cases produces a 500. A @RestControllerAdvice with ProblemDetail (native in Spring Boot 3.x) mapping domain exceptions to 404/409/422 is the minimum needed for a production-ready API. Pair this with @Valid on request bodies and spring-boot-starter-validation in the POM.
